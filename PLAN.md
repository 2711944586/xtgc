   无用内容，重写 README，并让提交包结构自包含、清晰、可检查。

## 1. 不可破坏的研究约束

升级过程中不得为了视觉效果牺牲研究可信度，以下约束保持不变：

- 数据来源为 U.S. DOT BTS Airline On-Time Performance 公开数据。
- 默认样本为 2024 年 1 月至 3 月，筛选样本内流量前 15 个机场。
- 预测目标为 `ArrDel15`，即到达延误是否达到 15 分钟。
- 训练、验证、测试必须采用时间顺序划分，不能随机划分。
- 计划阶段预测不得使用实际起降时间、到达延误分钟数、延误原因等泄漏字段。
- SHAP、ISM、传播矩阵和仿真结果只能表述为结构性证据或关联推断，不能夸大为严格因果。
- 四类恢复策略必须在相同情景、相同预算口径、相同初始状态下比较。
- 成本、容量下降、恢复资源等参数属于透明的相对情景设定，不能伪装成真实企业成本。
- `dynamic_combo` 是主偏好下的推荐，不是无条件最优；必须保留风险/成本保守条件下的排名反转说明。
- 报告、PPT、讲稿、Web Demo、README 和提交包中的核心数字必须一致。

## 3. 静态 Web Demo 升级计划

### 3.1 目标定位

Web Demo 不再只是“附带页面”，而是答辩现场的交互式证据面板。第一屏直接进入
研究对象和关键结论，后续模块按“数据规律 -> 预测输入 -> 网络结构 -> 扰动仿真 ->
策略评价 -> 方法说明”的路径组织。

### 3.2 必须实现的模块

1. **总览与关键结论**
   - 样本范围、航班量、机场数、主推荐策略。
   - 研究流程条：真实数据、风险预测、网络识别、传播仿真、策略评价、静态部署。

2. **数据驾驶舱**
   - 每日航班量与延误率趋势。
   - 星期-小时延误热力图。
   - 机场延误率与风险排名。
   - 支持机场选择并联动其他模块。

3. **延误预测解释**
   - 快速风险估算器：起降机场、小时、距离、拥堵水平。
   - 模型指标对比。
   - SHAP/特征重要性条形图。
   - 明确说明这是计划阶段演示估算，不替代在线调度系统。

4. **机场网络与关键节点**
   - Canvas 或 SVG 网络图，节点大小/颜色体现流量和关键性。
   - 机场画像：航班量、延误率、预测风险、介数中心性、关键性。
   - 关键机场排名条形图。

5. **扰动仿真与策略对比**
   - 情景切换：normal、peak、weather、hub_capacity_drop。
   - 策略切换：baseline、uniform_buffer、hub_priority、dynamic_combo。
   - 恢复曲线：性能、累计延误、成本或关键指标。
   - 策略指标卡：恢复时间、最低性能、传播范围、取消代理、相对成本。

6. **综合评价与敏感性**
   - TOPSIS 综合得分。
   - AHP/熵权组合权重敏感性滑杆。
   - 风险型决策期望损失表。
   - 主动呈现“dynamic_combo 有条件推荐、baseline 在保守条件下可能反超”。

7. **方法与复现说明**
   - 核心公式：预测概率、网络权重、状态传播、TOPSIS 接近度。
   - 数据来源和静态资产导出说明。
   - 部署路径和仓库结构提示。

### 3.3 视觉与交互标准

- 使用航空运行控制中心风格：浅底、深墨蓝文字、航空橙风险强调、青绿恢复强调。
- 避免紫蓝渐变、过度圆角、模板卡片堆叠和无意义装饰。
- 图表类型多样：折线、热力、条形、散点、网络、矩阵、表格、仪表/进度条。
- 控件要有真实联动：选择机场、策略、情景、权重后图表和解释文本随之更新。
- 移动端、平板和桌面端都不得出现文字溢出、按钮拥挤或横向滚动失控。
- 必须提供加载失败状态，避免 JSON 缺失时空白。

### 3.4 静态部署要求

- `web/index.html`、`web/assets/css/styles.css`、`web/assets/js/app.js`、
  `web/assets/data/flightresilience-data.json` 为核心静态产物。
- 提供 `web/README.md`，说明本地预览与 GitHub Pages 发布方式。
- 提供 `.github/workflows/pages.yml`，默认将 `web/` 作为 Pages artifact 上传。
- README 写清目标仓库 `2711944586/xtgc` 的推荐部署路径。
- 若当前目录不是 git 仓库，不强行推送；只保证静态产物和说明完备。

### 3.5 现场演示路径

为了避免答辩时“打开页面但不知道点什么”，Web Demo 必须内置一条清晰演示线：

1. 首页确认样本范围、整体延误率和主推荐策略。
2. 数据驾驶舱选择 DEN 或其他关键机场，展示延误率、热力图和机场排名联动。
3. 预测模块调节小时、距离和拥堵水平，说明计划阶段风险概率如何变化。
4. 网络模块切换机场画像，说明高流量不等于最高传播关键性。
5. 仿真模块选择天气或枢纽容量下降情景，比较四策略恢复曲线。
6. 决策模块拖动偏好权重，主动展示 dynamic_combo 推荐条件和 baseline 反超条件。
7. 方法模块说明公式、数据来源、静态部署和结论边界。

## 4. 报告升级计划

### 4.1 报告结构

报告定位为可独立阅读的系统工程研究报告，目标篇幅显著高于当前简版。建议结构：

1. 摘要、关键词、研究贡献。
2. 绪论：问题背景、研究意义、为什么单航班预测不足。
3. 系统工程问题定义：系统边界、5W1H、利益相关者、目标树、评价指标体系。
4. 方法体系：霍尔三维结构、切克兰德软系统、鱼骨图、ISM、机器学习、复杂网络、
   状态空间仿真、AHP/熵权/TOPSIS/模糊评价/风险决策。
5. 数据来源与数据审计：BTS 来源、字段、样本、时间划分、防泄漏规则。
6. 探索性分析：时间异质性、机场差异、航班量与延误率关系。
7. 预测模型：特征工程、模型对比、指标解释、校准、SHAP。
8. 机场网络：有向加权网络、中心性、关键节点、网络风险解释。
9. ISM 结构分析：邻接矩阵、可达矩阵、层级划分、管理含义。
10. 传播仿真：状态变量、传播矩阵、冲击项、控制项、模型验证。
11. 策略评价：四策略、多情景、多指标、TOPSIS、模糊评价、风险/不确定决策。
12. 敏感性与结论边界：主推荐、反转条件、适用场景。
13. 静态 Web Demo：页面结构、交互路径、部署方式、截图说明。
14. 结论、局限与个人实践体会。
15. 附录：数据字典、参数表、AHP 判断矩阵、ISM 矩阵、传播矩阵、核心代码索引。

### 4.2 必须补充的公式

报告至少包含以下公式，并在正文解释变量含义：

- 计划阶段延误概率：`p_i=P(y_i=1|x_i)=f(x_i)`。
- 航线权重：`w_{ij}=alpha F_{ij}+beta R_{ij}+gamma D_{ij}`。
- 机场状态传播：`x(t+1)=A x(t)+B u(t)+G w(t)+epsilon(t)`。
- 综合关键性：`K_i=omega_1 flow_i+omega_2 betweenness_i+omega_3 risk_i+omega_4 delay_i`。
- 熵权：`e_j=-k sum_i p_{ij} ln(p_{ij})`，`w_j=(1-e_j)/sum_j(1-e_j)`。
- TOPSIS 接近度：`C_i=D_i^-/(D_i^+ + D_i^-)`。
- 风险型期望损失：`E(L_s)=sum_c P(c)L_{s,c}`。

### 4.3 必须补充的代码片段

代码片段不追求长，而要展示关键技术步骤：

## 5. PPT 升级计划

### 5.1 技能与模式

- 使用 presentations skill。
- 任务模式为 `create` 或基于既有内容的显著重构，不按模板克隆。
- 主 profile 选 `engineering-platform`，次级约束为 `strategy-leadership` 和数据分析叙事。
- PPT 必须使用 claim spine、design system、contact sheet plan 和预览检查。

### 5.2 叙事主线

```text
局部延误不是孤立事件
  -> 航空网络会放大扰动
  -> 真实数据揭示风险规律
  -> 预测模型提供计划阶段风险输入
  -> 网络和 ISM 识别关键传播结构
  -> 状态空间仿真比较恢复策略
  -> 多准则评价给出有条件推荐
  -> 静态 Web Demo 支撑现场演示与复现
```

### 5.3 建议页序

1. 封面：一个枢纽延误会变成全网恢复问题。
2. 问题缺口：单航班预测不能回答传播和恢复资源分配。
3. 系统边界：主体、外部环境、目标冲突。
4. 方法链：霍尔 + 切克兰德 + 数据/网络/仿真/评价。
5. 数据证据：BTS 样本、时段热力、机场差异。
6. ISM：根源因素到恢复时间的结构链。
7. 预测：时间外模型指标与 SHAP 解释。
8. 网络：关键节点与传播位置。
9. 仿真：状态方程、传播矩阵、恢复曲线。
10. Web Demo：现场演示路径和截图。
11. 决策：TOPSIS、敏感性、风险条件下排名反转。
12. 结论：贡献、局限、个人思考。

如报告内容进一步扩展，PPT 可增加 1-3 页附录型证据页，但主讲页保持 12 页左右。

### 5.4 PPT 质量标准

- 每页标题必须是判断句，不使用“模型介绍”“结果分析”等目录式标题。
- 每页有一个 dominant proof object：图、表、流程、公式或截图。
- 信息密度高但不拥挤，缩略图能看出不同版式节奏。
- 不使用无来源 Logo、伪品牌图标或装饰性模板元素。
- 关键数字与报告、Web 一致。
- 生成后检查 contact sheet 和单页 PNG，修复文字溢出、图表遮挡和低对比问题。

## 6. 讲稿升级计划

讲稿与 PPT 页序一一对应，每页包含：

- 建议时间。
- 核心句。
- 详细讲述稿。
- 过渡句。
- 可删减句。
- Demo/网络异常备用说明。

总时长目标：7 分 30 秒左右。用户语速快，因此正文可以适当充实，但要标注可删减
内容，便于排练时压缩。

讲稿必须体现个人思考：

- 为什么把延误问题从预测扩展到网络恢复。
- 为什么采用时间切分和防泄漏策略。
- 为什么 `dynamic_combo` 不能说成绝对最优。
- 静态 Web Demo 在系统工程课程中的价值。
- 成本、容量、恢复资源假设带来的局限。

## 6.1 PPT 与讲稿精华抽取规则

报告会保留完整技术链路，但 PPT 和讲稿只抽取最能支撑答辩的精华：

- **从报告抽到 PPT**：只抽“问题缺口、方法链、关键证据、策略排名、反转边界、个人思考”，不把报告正文压缩成密密麻麻的文字页。
- **从 Web 抽到 PPT**：只放首页、仿真、决策三类高价值截图，截图必须能说明现场交互路径。
- **从图表抽到 PPT**：优先热力图、网络图、传播曲线、TOPSIS/敏感性图，避免重复堆叠普通柱状图。
- **从 PPT 扩到讲稿**：每页讲稿先给一句判断，再补数据依据和管理含义；复杂公式在讲稿中只解释变量和用途。
- **从讲稿保留个人表达**：讲清楚为什么不只做预测、为什么坚持时间切分、为什么推荐不能说成绝对最优、为什么做静态 Web。
